 從這個專案的開發觀察到，每一次的調整很容易把舊有的部分沒有要調整代碼丟失，要如何避免這種狀況?

Gemini 說了
這是一個非常深刻的開發痛點。在「AI 協作開發」中，最常遇到的問題就是：AI 為了給你「最新、最簡潔」的代碼，往往會自動「修剪」掉它認為不重要、但對你而言卻是核心功能的舊代碼。

這在軟體工程中稱為「回歸錯誤 (Regression)」。要避免你的 MyAiTool 越改功能越少，我建議 Jason 你可以建立以下幾道防線：
1. 核心邏輯「工具化」 (Utility Decoupling)

這是你已經在做的（規則 2：拆分檔案），但可以更徹底。

    做法：不要把「功能邏輯」寫在 gui_manager.py 或 downloader.py 這種大型檔案裡。
    優點：當我們修改介面時，只要 utils/file_tool.py 或 utils/logger_tool.py 沒動，核心的讀寫與日誌邏輯就不會丟失。
    對 AI 的指令：每次請我修改時，明確要求：「請保持原本 _insert_to_tree 函數的內部邏輯不變，僅調整 update_ui_components 的過濾條件。」

2. 建立「功能清單 (Checklist)」

在你的 專案自動審計報告 或 功能說明書.md 中，列出每個模組的「不可丟失功能」。
    例如 failed_viewer.py：
        瀏覽器開啟 (webbrowser)
        複製連結 (pyperclip)
        設回 Pending (status update)
        開啟 Logs 資料夾 (os.startfile)

    開發習慣：每次貼出新代碼給我看之前，先問我：「請對照功能清單，確認這份新代碼是否保留了『複製連結』與『開啟 Logs』功能？」

3. 使用版本控制 (Git) —— 最強防線
對於 8GB RAM 的電腦來說，安裝一個輕量級的 Git 是最划算的投資。
    做法：每次我給你一段代碼，你貼上並測試成功後，執行 git commit。
    救命關鍵：萬一發現這次調整後「右鍵選單不見了」，你只要輸入 git diff 就能立刻看到哪幾行被 AI 刪掉了，然後手動補回來。

4. 要求「部分修改」而非「全量覆蓋」

如果你覺得檔案太大，全量覆蓋很危險，可以對我下達更精準的指令：
    「我只需要修改 update_ui_components 函數裡的 for 迴圈過濾邏輯，其餘函數請保持原樣，並只給我修改後的那段代碼即可。」
