管理 AI 專案的程式碼與傳統軟體開發既有相似之處，也有其獨特的挑戰。因為 AI 專案不僅包含程式碼，還涉及大量的數據 (Data) 和模型權限 (Models)。

以下是目前業界管理 AI 專案的主流做法：
1. 程式碼版本控制：Git 體系

對於單純的程式碼（如模型架構定義、訓練腳本、預處理邏輯），我們依然使用傳統的 Git 進行管理。

    平台： GitHub, GitLab, 或 Bitbucket。

    管理重點：

        環境配置文件： 使用 requirements.txt 或 environment.yml (Conda) 記錄依賴套件。

        實驗腳本： 記錄不同版本的超參數（Hyperparameters）設定。

2. 數據與大型檔案管理：DVC (Data Version Control)

Git 不適合處理 GB 等級的大數據集或模型權重文件。為了補足這一點，開發者通常會搭配 DVC。

    核心功能： DVC 就像是「數據界的 Git」。它不會把大檔案塞進 Git，而是將檔案上傳到雲端儲存（如 S3, Google Cloud Storage），並在 Git 中只留下一個很小的 .dvc 指標文件。

    優點： 確保你的程式碼版本與當時使用的數據集版本是完全對齊的。

3. 實驗追蹤與模型管理：MLOps 工具

在 AI 開發中，你可能會跑幾百次實驗，每次的準確度（Accuracy）和損失函數（Loss）都不同。手動記錄在 Excel 是很低效的。

    常見工具： MLflow, Weights & Biases (W&B), 或 Comet.ml。

    管理內容：

        指標追蹤： 自動繪製訓練曲線。

        模型註冊表 (Model Registry)： 標記哪個版本的模型是「生產中 (Production)」，哪個是「開發中 (Staging)」。

        重現性： 記錄訓練時的硬體環境、隨機種子（Random Seed）等。

4. 環境容器化：Docker

AI 專案對硬體驅動（如 CUDA, cuDNN）和套件版本非常敏感。

    做法： 將開發環境打包成 Docker Image。

    目的： 確保「在我的電腦能跑，在伺服器上也能跑」，避免因環境差異導致模型推理結果不一致。

5. 典型的專案目錄結構

一個專業的 AI 專案通常會長這樣：
Plaintext

├── data/               # 原始數據與處理後的數據 (通常會被 .gitignore 忽略)
├── models/             # 存放訓練好的模型權重 (.pth, .onnx)
├── notebooks/          # Jupyter Notebooks (用於初步實驗與視覺化)
├── src/                # 核心原始碼
│   ├── train.py        # 訓練腳本
│   ├── evaluate.py     # 評估腳本
│   └── preprocess.py   # 數據清洗
├── tests/              # 測試代碼
├── requirements.txt    # 套件依賴
└── dvc.yaml            # DVC 配置檔

總結：AI 程式碼管理的金三角
類別	管理工具	管理對象
Code	Git / GitHub	演算法、訓練邏輯、設定檔
Data	DVC / LakeFS	訓練集、驗證集、標籤
Experiment	MLflow / W&B	超參數、精準度、模型權重
